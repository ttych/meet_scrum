// # What is Scrum ?

// Scrum is a **framework** for developing and sustaining **complex**
// products.

// Scrum is composed of :
// - **rules**
// - **3 roles**
// - **5 human values**
// - **4 events**
// - **3 pillars**
// - **3 artifacts**

// In short, Scrum is :
// - **lightweight** framework
// - **simple** to understand
// - **hard** to master




// # Principles

// Purpose is to generate value through adaptive solutions for complex
// problems.

// Scrum is founded on
// - **empiricism** : knowledge comes from experience
// - **lean thinking** : reduce waste and focuses on the essentials

// Empiricism :
// - based on past experience
// - short development cycle, Sprint : iterative, incremental
// - inspect and adapt

// Lean Thninking :
// - minimal
// - sprint contains 4 events focused on inspection and adaptation
// - purposefully incomplete: wrapper around practices, to add on need

// # Scrum Overview

// ![Scrum Framework](https://nkdagility.com/wp-content/uploads/2020/11/naked-Agility-Scrum-Framework.jpg)

// # Rules

// To guide people relationships and interactions :
// - A **Product Owner** (role 1) orders the work for a complex problem into a
//   Product Backlog.
// - The **Scrum Team** (role 2+3) turns a selection of the work into an Increment of
//   value during a Sprint
// - The **Scrum Team** (role 2+3) and its stakeholders inspect the results and adjust
//   for the next Sprint
// - Repeat


// # 3 Roles

// Each role has it own accountability.

// ## The Scrum Team : Product Owner + Scrum Master + Developers

// A small (<= 10) team of people composed of :
// - 1 scrum master
// - 1 product owner
// - developers

// Smaller teams communicate better and are more productive.

// **Organized :**
// - no sub-teams.
// - no hierarchies.
// - as a cohesive unit.
// - focused on the Product Goal.
// - self-managing, meaning they internally decide who does what, when, and how.
// - cross-functional.
// - working at a sustainable pace.

// **Responsible for :**
// - all product-related activities from stakeholder collaboration,
//   verification, maintenance, operation, experimentation, research and
//   development, and anything else that might be required

// **Accountable for :**
// - creating a valuable, useful Increment every Sprint
// - inspecting the results and adapting for the next Sprint

// ## Product Owner

// Product owner is 1 person, representing many stakeholders.

// **Accountable for effective product backlog management :**
// - Developing and explicitly communicating the Product Goal
// - Creating and clearly communicating Product Backlog items
// - Ordering Product Backlog items
// - Ensuring that the Product Backlog is transparent, visible and
//   understood.

// The entire organization must respect the Product Owner decisions.

// ## Developers

// Developers are committed to create usable Increment each Sprint.

// **Accountable for :**
// - creating a plan for the Sprint, the Sprint Backlog
// - instilling quality by adhering to a Definition of Done
// - adapting their plan each day toward the Sprint Goal
// - holding each other accountable as professionals

// ## Scrum Master

// Help understand Scrum in the team and in the organization.

// **Accountable for :**
// - establishing Scrum as defined in the Scrum Guide
// - the Scrum Team's effectiveness

// The Scrum master is a Servant-Leader for the Scrum Team and the larger
// organization.

// **Serving the team by :**
// - Coaching the team members in self-management and cross-functionality
// - Helping the Scrum Team focus on creating high-value Increments that
// meet the Definition of Done
// - Causing the removal of impediments to the Scrum Teamâ€™s progress
// - Ensuring that all Scrum events take place and are positive,
//   productive, and kept within the time-box.

// **Serving the product owner by :**
// - Helping find techniques for effective Product Goal definition and
// Product Backlog management
// - Helping the Scrum Team understand the need for clear and concise
// Product Backlog items
// - Helping establish empirical product planning for a complex
// environment
// - Facilitating stakeholder collaboration as requested or needed

// **Serving the organization by :**
// - Leading, training, and coaching the organization in its Scrum
// adoption
// - Planning and advising Scrum implementations within the organization
// - Helping employees and stakeholders understand and enact an empirical
// approach for complex work
// - Removing barriers between stakeholders and Scrum Teams


// # 5 Values

// Rely on 5 people values:
// - Commitment: to its goals and supporting each other
// - Focus: 1st focus is on the sprint content to make the best possible
//   progress toward these goals
// - Openness: open on the work and challenges
// - Respect: each other (people are capable & independent)
// - Courage: to do the right thing, work on tough problems

// Allow a mindset for a successful use of Scrum.


// # 4 Events

// Sprint is a container for all other events.

// Events are :
// - an opportunity to inspect and adapt Scrum artifacts.
// - used to create regularity.
// - minimize the needs for other meetings.
// - designed to provoke change.

// Optimally, all events are held at the same time and place to reduce
// complexity.

// ## Sprint

// Is :
// - Fixed length events (<= 1 month)
// - repeatable
// - a learning cycle

// Contains the following events, in the following order :
// - Sprint Planning
// - Daily Scrum
// - Sprint Review
// - Sprint Retrospective

// Sprint rules are:
// - No changes are made that would endanger the Sprint Goal
// - Quality does not decrease
// - The Product Backlog is refined as needed
// - Scope may be clarified and renegotiated with the Product Owner as
//   more is learned.

// A Sprint could be canceled if the Sprint Goal becomes obsolete.
// Only the Product Owner has the authority to cancel the Sprint.

// Various practices exist to forecast progress, like burn-downs, burn-ups, or cumulative flows.

// ## Sprint Planning

// Purpose:
// - have a plan for the current sprint

// Who:
// - Scrum Team

// When:
// - it initiates the Sprint.

// Preparation:
// - prepare most important Product Backlog items (how they map to the
//   Product Goal)
// - team knows about their velocity, their upcoming capacity

// What:
// - Why is this Sprint valuable? =>
//   - Product Owner: proposes how the product could increase its value and utility
//   - Scrum Team: define the Sprint Goal
// - What can be Done this Sprint?
//   - Scrum Team: refine backlog item to increase understanding and confidence
// - How will the chosen work get done?
//   - Developers: create increment that meets DOD
//   - Developers: decomposing Product Backlog items into smaller work
//     items of one day or less

// Length:
// - 8 hours for 4 weeks sprint
// - 4 hours for 2 weeks sprint

// Outcome:
// - the Sprint Backlog: a plan created by the collaborative work of the entire Scrum Team.
// - the Sprint Backlog = Sprint Goal + the Product Backlog items selected for the Sprint.

// ## Daily Scrum

// Purpose:
// - improve communication
// - identify impediments
// - promote quick decision-making (eliminate the need for other
//   meetings)
// - adapt the Sprint Backlog as necessary

// Who:
// - Developers

// When:
// - every day
// - at the same time, at the same place (to reduce complexity)

// Outcome:
// - adjusted sprint backlog
// - adjusted upcomming planned work

// What:
// - focus on progress toward the sprint goal
// - action plan for the next day of work
// - inspect progress and adapt sprint backlog

// Length:
// - 15 min (max.)

// ## Sprint Review

// Purpose:
// - inspect the outcome of the sprint (the what)
// - determine future product adaptations

// Who:
// - Scrum Team
// - Stakeholders

// When:
// - the second to last event of the Sprint

// Outcome:
// - what to do next
// - adjusted product backlog

// What:
// - do a presentation
// - collect feedback

// Length:
// - 4 hours for 4 weeks sprint
// - 2 hours for 2 weeks sprint

// ## Sprint Retrospective

// Purpose:
// - plan ways to increase quality and effectiveness (the how)

// Who:
// - Scrum Team

// When:
// - conclude the Sprint

// Outcome:
// - identify the most helpful changes to improve its effectiveness
// - the most impactful improvements are addressed as soon as possible
//   (sprint backlog)

// What:
// - inspect the last Sprint
// - how it went with regards to individuals, interactions, processes,
//   tools, and their Definition of Done
// - what went well ? which problems encountered ? how those problems
//   were (or were not) solved ?

// Length:
// - 3 hours for 4 weeks sprint
// - 1 hours 30 for 2 weeks sprint


// # 3 Pillars

// # Scrum pillars

// The empirical Scrum pillars are transparency, inspection, and
// adaptation.

// Transparency -> Inspection -> Adaptation :
// - Transparency enables inspection.
// - Inspection enables adaptation.

// Inspection without transparency is misleading and wasteful.

// Inspection without adaptation is considered pointless.

// ## Transparency

// Transparency in :
// - process.
// - work.
// - artifacts.

// Because low transparency :
// - diminish outcome values of inspection
// - can lead to decisions that diminish value and increase risk.

// ## Inspection (Learn)

// Inspect frequently to detect potentially undesirable variances or
// problems.

// At least 4 events to inspect !

// ## Adaptation (Adjust)

// Must be made as soon as possible to minimize further deviation.


// # 3 Artifacts

// Scrum's artifacts represent work or value.
// They are designed to maximize transparency of key information and
// focus.

// Each artifact contains a commitment :
// - for the Product Backlog it is the Product Goal
// - for the Sprint Backlog it is the Sprint Goal
// - for the Increment it is the Definition of Done

// ## Product Backlog

// The Product Backlog is an emergent, ordered list of what is needed to
// improve the product.
// It is the single source of work undertaken by the Scrum Team.

// Product Backlog refinement is the act of breaking down and further
// defining Product Backlog items into smaller more precise items.

// Refinement :
// - ongoing activity
// - add details: description, order, size

// Commitment : The Product Goal
// - transparency
// - long-term objective for the Scrum Team
// - must fulfill (or abandon) one objective before taking on the next

// > A product is a vehicle to deliver value. It has a clear boundary,
// > known stakeholders, well-defined users or customers. A product could
// > be a service, a physical product, or something more abstract.

// ## Sprint Backlog

// It is composed of :
// - the Sprint Goal (why)
// - the selected Product Backlog items (what)
// - an actionable plan for delivering the Increment (how)

// It is :
// - a plan by and for the Developers
// - highly visible, real-time picture of the work

// Commitment : The Sprint Goal
// - transparency
// - creates coherence and focus
// - encouraging the Scrum Team to work together rather than on separate
//   initiatives
// - provides flexibility in terms of the exact work needed to achieve it

// ## Increment

// It is :
// - a concrete stepping stone toward the Product Goal
// - additive to all prior Increments (incremental)
// - verified (all Increments work together)
// - usable
// - delivered to stakeholders
// - meets the Definition of Done

// Commitment : The Definition of Done
// - transparency
// - formal description of the increment of the state of the Increment
//   when it meets the quality measures required for the product

// > The moment a Product Backlog item meets the Definition of Done,
// > an Increment is born.


// # To sum up

// - Transparency allows Inspection which allows Adaptation.
// - Values to allow Transparency.
// - Transparency in Artifacts
// - Transparency in Artifact's commitment.
// - Transparency, Inspection, Adaptation in all Events.
// - Focus on goal, the what more than sticking to the plan, the how
